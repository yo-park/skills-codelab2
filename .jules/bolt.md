## 2024-04-20 - Removed String Allocations From Scanner Hot Path
**Learning:** In the inner loop of `ScanEngine::process_file_stream`, literal keyword searches were allocating string clones and performing `to_lowercase` on both the input string AND every single search keyword for every single line. This O(N * M) allocation overhead on the hot path caused substantial and unnecessary memory pressure.
**Action:** Always verify if keyword pre-computation or case conversions can be hoisted out of a tight IO loop, especially in a fast language like Rust, and avoid mapping or cloning inside hot read loops unless strictly required by mutation semantics.
## 2024-05-04 - Record vs Map for high frequency Zustand updates
**Learning:** In a high-frequency SSE event listener handling hundreds of updates per second, cloning an ES6 `Map` (`new Map(state)`) inside a Zustand state setter creates enormous GC churn and drops frames. Switching to a standard JS object (`Record`) and using object spread (`...state`) benchmarked at ~12ms vs ~902ms for 100k iterations. Furthermore, returning a new state reference when an item isn't found causes unnecessary React re-renders; always return the unmodified state reference if no data changed.
**Action:** Always prefer plain objects (`Record`) over ES6 `Map`s for dictionaries in React state managers (like Zustand or Redux) to leverage structural sharing, minimize allocation, and prevent useless re-renders.
