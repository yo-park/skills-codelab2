## 2024-04-20 - Removed String Allocations From Scanner Hot Path
**Learning:** In the inner loop of `ScanEngine::process_file_stream`, literal keyword searches were allocating string clones and performing `to_lowercase` on both the input string AND every single search keyword for every single line. This O(N * M) allocation overhead on the hot path caused substantial and unnecessary memory pressure.
**Action:** Always verify if keyword pre-computation or case conversions can be hoisted out of a tight IO loop, especially in a fast language like Rust, and avoid mapping or cloning inside hot read loops unless strictly required by mutation semantics.

## 2026-05-01 - Zustand State Throttling and Map Usage on High-Frequency Streams
**Learning:** Using an ES6 `Map` in a Zustand store requires creating a new Map instance for partial state updates, which causes React components that depend on it to rapidly re-render. Combining this with high-frequency Server-Sent Events (SSE) resulted in significant frontend lag. Plain JS object (`Record`) manipulation allows localized, lightweight updates, and chunking these updates inside a short buffer via `setTimeout` throttles the React rendering cycle to a manageable interval without losing final state consistency (when buffers are explicitly flushed on stream closure).
**Action:** Always prefer plain objects (`Record`) over ES6 `Map`s for state inside Zustand, particularly when items receive rapid individual updates. When consuming high-frequency event streams (like SSE progress logs), always debounce or throttle the state mutations using a timer and a buffer pattern, being careful to force a flush when the stream errors or concludes.
